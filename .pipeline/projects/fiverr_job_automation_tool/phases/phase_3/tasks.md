# Phase 3 Tasks

- [ ] Task 1: Build opportunity scorer module
  - What: Create a scoring engine that evaluates scraped job opportunities based on keyword match, budget fit, and buyer credibility. Score each job on a 0-100 scale.
  - Files: 
    - Create `src/scorer.py` — `OpportunityScorer` class with methods: `score(job)`, `score_keyword_match(job, keywords)`, `score_budget_fit(job, min_budget)`, `score_buyer_rating(buyer_rating)`, `compute_overall_score(job, keywords, min_budget)`
    - Create `src/models.py` — `JobOpportunity` dataclass with fields: `id`, `title`, `description`, `budget_min`, `budget_max`, `deadline`, `buyer_name`, `buyer_rating`, `posted_time`, `keywords`, `score`
  - Done when: 
    - `OpportunityScorer.score()` returns a float 0-100 for any valid `JobOpportunity`
    - Keyword scoring gives higher scores when more keywords match (weighted by frequency)
    - Budget scoring gives 0 if job budget < min_budget, otherwise proportional score
    - Buyer rating scoring maps 0-5 star rating to 0-100 scale
    - Unit tests in `tests/test_scorer.py` verify scoring logic with mock jobs (≥5 test cases)

- [ ] Task 2: Build proposal template engine
  - What: Create a template-based proposal generator that fills in job-specific variables (client name, job title, keywords, budget) into configurable proposal templates.
  - Files:
    - Create `src/proposal.py` — `ProposalEngine` class with methods: `load_templates(template_dict)`, `generate(job, template_name)`, `render(template, variables)`, `get_available_templates()`
    - Templates stored as dict in config or as YAML files under `templates/` directory
    - Variable substitution supports: `{client_name}`, `{job_title}`, `{job_description}`, `{budget}`, `{keywords}`, `{seller_name}`, `{custom_message}`
  - Done when:
    - `ProposalEngine.generate()` returns a formatted proposal string with all variables substituted
    - Missing variables raise a clear `ValueError` with the list of missing keys
    - Default templates included (e.g., "professional", "friendly", "short")
    - Unit tests in `tests/test_proposal.py` verify template rendering (≥4 test cases)

- [ ] Task 3: Build submission engine with dry-run mode and CSV logging
  - What: Create the bid submission engine that fills proposal forms (via Playwright or mock), supports dry-run mode, and logs all submissions to CSV.
  - Files:
    - Create `src/submission.py` — `BidSubmissionEngine` class with methods: `submit(job, proposal)`, `submit_dry_run(job, proposal)`, `log_submission(job, proposal, status)`, `get_submission_log()`
    - Create `src/submitter_playwright.py` — Playwright-based form filler: `fill_bid_form(job, proposal, page)` that navigates to the bid page, fills title/description fields, and clicks submit
    - Create `submission_log.csv` via `csv` module — columns: `timestamp, job_id, job_title, budget, score, proposal_preview, status, mode`
    - Dry-run mode skips actual submission but logs to CSV with status="DRY_RUN"
    - Rate limiting: configurable delay between submissions (default 30s)
  - Done when:
    - `BidSubmissionEngine.submit_dry_run()` logs a CSV row without making any browser/API calls
    - `BidSubmissionEngine.submit()` with Playwright fills and submits a bid form
    - CSV log file is created/updated with each submission
    - Rate limiting enforced via configurable delay
    - Unit tests in `tests/test_submission.py` verify dry-run logging and CSV output (≥4 test cases)

- [ ] Task 4: Add CLI entry point with flags and integrate scorer + proposal + submission
  - What: Create a CLI entry point (`main.py`) that ties together scraping (from Phase 2), scoring, proposal generation, and submission. Support CLI flags for configuration.
  - Files:
    - Create `main.py` — CLI entry point using `argparse` with flags: `--dry-run`, `--max-bids`, `--min-budget`, `--keywords` (comma-separated), `--template`, `--log-file`, `--output-dir`
    - Create `src/pipeline.py` — `AutomationPipeline` class that orchestrates: `scrape_jobs() → score_jobs() → generate_proposals() → submit_bids()`
    - Pipeline reads config from `config.py` and CLI args, runs the full flow
    - CLI help text explains each flag
    - `--dry-run` flag disables actual submissions, only logs what would be submitted
    - `--max-bids` limits number of bids submitted
    - `--min-budget` filters jobs below budget threshold
    - `--keywords` filters jobs by keyword match
  - Done when:
    - `python main.py --dry-run --max-bids 5 --min-budget 50 --keywords "python,django"` runs without error
    - Pipeline processes jobs through all stages and stops at max-bids
    - Dry-run mode produces CSV log without submitting
    - CLI `--help` shows all flags with descriptions
    - Integration test in `tests/test_pipeline.py` verifies end-to-end flow (≥3 test cases)

- [ ] Task 5: Write end-to-end integration test and validation
  - What: Write a comprehensive integration test that exercises the full pipeline with mock data, and validate all Phase 3 deliverables.
  - Files:
    - Create `tests/test_integration.py` — Full pipeline integration test using mocked scraper output, scoring, proposal generation, and dry-run submission
    - Create `tests/fixtures/mock_jobs.py` — Mock job data fixtures for testing
    - Test verifies: jobs are scored correctly → proposals are generated → dry-run submissions are logged → CSV output is valid
    - Add `pytest` markers for integration tests (`@pytest.mark.integration`)
  - Done when:
    - `pytest tests/test_integration.py` passes all tests
    - `pytest tests/` passes all Phase 3 tests (scorer, proposal, submission, pipeline)
    - Full pipeline runs end-to-end with mock data producing valid CSV output
    - No regressions in Phase 1 tests
